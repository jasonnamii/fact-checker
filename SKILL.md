# Fact Checker

문서의 팩트를 자동 추출하고 삼각측량으로 검증한 뒤, 수정안까지 생성한다.

---

## 절대 규칙

| # | 규칙 | 이유 |
|---|------|------|
| 1 | **수치는 Python 산술 검증 필수** — "대충 맞다" 판정 금지. 분자/분모 나눗셈, 합산 대조 | LLM은 근접 수치를 동일시하는 구조적 취약점 보유 (97.94 vs 98.26) |
| 2 | **소스 등급 판정 필수** — 공시>업종지>일반지>위키. 등급 없는 "다수 보도 일치" 금지 | 2차 소스끼리 교차하면 삼각측량이 아니라 순환 참조 |
| 3 | **검증불가 팩트는 🔍 표기** — 자사 정보 등 외부 검증 불가 시 "확인 불가" 명시 | 검증 안 된 팩트를 ✅로 넘기면 신뢰 붕괴 |
| 4 | **LLM 취약 영역 강제 스캔** — 괄호 안 수치, 표 내부, 부연설명 속 수치를 별도 패스로 재스캔 | LLM attention이 분산되는 위치에서 누락률 최고 |

## 모드 판정

| 호출 | 모드 | 범위 |
|------|------|------|
| "라이트 팩트체크" / "숫자만 확인" | **LIGHT** | 수치형 팩트만 (금액·비율·일자·규모) |
| "팩트체크해줘" / "딥 팩트체크" / 기본 | **DEEP** | 전수 (수치+관계+속성+인과+부재증거) |

---

## 실행 흐름

```
① 팩트 추출 → ② 검증 (4축 병렬) → ③ 판정·수정안 → ④ 보고 → [형 컨펌] → ⑤ 수정 적용
```

### ① 팩트 추출 (5W1H 프레임)

문서를 읽고 **5W1H 축**으로 검증 대상 팩트를 빠짐없이 추출한다. 맥락 기반 추출이므로 "형태"가 아니라 "역할"로 분류 — 문서의 주장이 성립하려면 6축 모두 맞아야 한다.

| 축 | 추출 대상 | 예시 | LIGHT | DEEP |
|----|-----------|------|-------|------|
| **Who** | 주체·기관·인물 | 두나무, 송치형, 하이브 | | ✓ |
| **What** | 사건·행위·산출물 | 합병, 서비스종료, 리더십교체 | | ✓ |
| **When** | 시점·기간·일정 | 2025.11 발표, 2027.2 시행 | ✓ | ✓ |
| **Where** | 지리·범위·규모 | 200개국, 500㎡, 해외 77% | ✓ | ✓ |
| **How much** | 금액·비율·수량 | 1조5,578억, 98.26%, 404억 | ✓ | ✓ |
| **Why/How** | 인과·논리·근거 | "기술만으론 안 된다", "생존의 결과" | | ✓ |

**1패스 전수 + 조건부 2패스:** 5W1H 전수 추출을 1패스로 수행. 2패스(LLM 재스캔)는 1패스에서 불일치·모호·미검증으로 플래그된 항목만 대상. 전수 2패스 금지.

**LIGHT 모드:** When + Where + How much 3축만 추출·검증. Who/What/Why는 스킵.
**DEEP 모드:** 6축 전수 + 부재증거 탐지 (있어야 하는데 없는 팩트).

### ② 검증 (4축 병렬)

4개 축을 **병렬 WebSearch + Python**으로 실행한다.

| 축 | 핵심 | 상세 |
|----|------|------|
| 삼각측량 | 2+ 독립소스 교차, 소스등급 판정, 1차소스 추적 | → references/axis-triangulation.md 참조 |
| 연역 검증 | 내부정합성, 인과체인, 부재증거 탐지 | → references/axis-deduction.md 참조 |
| 단위·숫자 정합 | 단위변환, 합산기준, 시계열, Python 산술 | → references/axis-numerical.md 참조 |
| LLM 취약점 보정 | 근접숫자, 고유명사 스펙, 법률일자, 지표용어, 비율분모 | → references/llm-weakness.md 참조 |

**검색 전략:** 팩트 1개당 검증 쿼리 1~2개 생성. 독립 팩트는 WebSearch 병렬 발행 (최대 5개 동시). 관련 팩트는 1개 검색으로 묶음.

### ③ 판정 + 수정안

| 등급 | 기준 | 처리 |
|------|------|------|
| ✅ 정확 | 2+ 소스 일치, 산술 검증 통과 | 유지 |
| ⚠️ 미세오차 | 사실상 맞으나 반올림·표현 차이 | 용도별 판정 → references/tolerance.md 참조 |
| ❌ 수정필요 | 소스와 불일치, 산술 오류, 단위 혼동 | 수정안 생성 (Before → After) |
| 🔍 검증불가 | 자사 정보, 미래 예측, 소스 부재 | 명시 후 넘김 |

**수정안 생성:** ❌ 항목은 문서 내 해당 원문 + 수정 제안 + 근거 소스를 묶어 출력.

### ④ 보고

Python 스크립트로 검증 카드 + 요약 통계 출력:

```
[ID] {판정}  {문서 원문}
  실제: {검증 결과}
  소스: {소스명(등급)} — {URL}
  수정안: {Before → After} (❌만)
══════════════════
총 N개 | ✅ n | ⚠️ n | ❌ n | 🔍 n
```

### ⑤ 수정 적용 (형 컨펌 후)

❌ + ⚠️ 항목의 수정안을 **AskUserQuestion으로 일괄 제시** → 형 컨펌 → Edit으로 일괄 적용. EDIT4 연동:
- 수치 교체 (L1) → report+proceed
- 문장 재구성 (L2) → report+proceed
- 맥락 변경 (L3+) → Before/After 제시+confirm

적용 후 grep으로 잔여 확인 (구 수치 패턴 0건 검증).

## 예시: 실제 실행 흐름

```
형: "이 제안서 팩트체크해줘" + 파일 경로
→ DEEP 모드 발동
→ ① 5W1H 추출: Who 4건, What 5건, When 6건, Where 3건, How much 10건, Why 2건 = 총 16개 (중복 태깅 포함)
     2패스 재스캔: 괄호 안 수치 2건 추가 발견
→ ② WebSearch 5개 병렬 × 3라운드 = ~15회 검색
     Python 산술검증 (비율 분모/분자 나눗셈, 합산 대조)
→ ③ ✅9 ⚠️4 ❌3 🔍0 판정
→ ④ 검증 카드 출력 + 요약
→ ⑤ "다음 7건 수정 적용할까요?" → 형 컨펌 → Edit 일괄 적용 → grep 잔여 0건
```

## Gotchas

| 함정 | 대응 |
|------|------|
| **"대충 맞다" 스킵** | 97.94 vs 98.26처럼 근접 수치야말로 가장 위험. Python 필수 |
| **2차 소스 순환** | 언론A가 언론B를 인용 → 2개 소스가 아님. 원문(공시·IR)까지 추적 |
| **괄호 안 수치 누락** | LLM attention 분산. 추출 단계에서 괄호·표·부연을 별도 패스로 재스캔 |
| **검증불가를 ✅로 넘김** | 자사 정보(Cre8orClub 사양 등)는 외부 검증 불가. 🔍 명시 |
| **합산 기준 미확인** | 부문합산 vs 연결매출, 매출 vs 거래액(TTV). "큰 숫자=매출" 가정 금지 |
| **법률 일자 혼동** | 국회통과일 ≠ 공포일 ≠ 시행일. 3개를 구분하여 검증 |
| **반올림 무조건 OK** | 용도에 따라 다름. IR자료는 소수점 필수, 제안서는 반올림 허용. → tolerance.md |
| **부재 증거 미탐지** | 문서가 의도적으로 빠뜨린 불리한 팩트. 연역 검증에서 "있어야 하는데 없는 것" 탐색 |
