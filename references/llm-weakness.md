# LLM 취약점 보정 (LLM Weakness Compensation)

LLM이 구조적으로 놓치기 쉬운 7가지 영역. 팩트 추출(①)과 검증(②) 모두에서 적용.

---

## 7대 취약 영역

| # | 취약점 | 증상 | 보정 방법 |
|---|--------|------|-----------|
| 1 | **숫자 근접 혼동** | 97.94%를 98.26%와 "같다"고 처리 | Python 산술 필수. 소수점 이하까지 비교 |
| 2 | **고유명사 스펙 환각** | "500평 연습실, 800석"처럼 시설 규모를 잘못 기억 | 고유명사+숫자 조합은 반드시 검색 검증 |
| 3 | **법률·제도 일자 혼동** | 국회통과일·공포일·시행일을 섞음 | 3개 일자를 구분하여 각각 검증 |
| 4 | **지표 용어 혼동** | 매출/영업이익/순이익, 매출/거래액(TTV), 연결/별도 | 수치의 정확한 지표명 + 기준(연결/별도) 확인 |
| 5 | **비율의 분모 미추적** | "77%"의 분모가 매출인지 거래액인지 불명확 | 비율 등장 시 "무엇의 77%인가" 명시적 확인 |
| 6 | **시제·가정법 혼동** | "시행된다"(확정)와 "시행 예정이다"(미확정) 구분 실패 | 확정/예정/조건부를 구분하여 태깅 |
| 7 | **attention 분산 영역** | 괄호·표·부연·접힘(details) 내부 수치를 건너뜀 | 2패스 재스캔으로 보정 (아래 참조) |

## 2패스 재스캔 프로토콜

1패스(일반 읽기)에서 놓치기 쉬운 위치를 2패스에서 명시적으로 재방문.

| 위치 | 왜 놓치는가 | 재스캔 방법 |
|------|------------|------------|
| 괄호 () 안 | 부연 정보로 인식, attention 저하 | 정규식 `\(.*?\d+.*?\)` 패턴으로 수치 포함 괄호 추출 |
| 표(table) 셀 | 마크다운 표 파싱 시 셀 단위 attention 분산 | 표를 행 단위로 분해하여 셀별 팩트 추출 |
| 부연설명 (—, — 이후) | 대시 뒤 내용을 부차적으로 처리 | 대시 구문 별도 추출 |
| HTML details/summary | 접힌 콘텐츠를 스킵하는 경향 | details 태그 내부를 별도 추출 |
| 각주·미주 | 본문 대비 낮은 priority | 각주 영역 별도 스캔 |

## Gotchas

- **보정 과잉 주의**: 모든 괄호를 팩트로 추출하면 노이즈 폭발. "수치를 포함한 괄호"만 대상
- **시제 판정 ≠ 팩트 오류**: "시행 예정이다"가 현시점에서 이미 시행됐으면 시제 업데이트 필요. 하지만 문서 작성 시점 기준으로는 정확했을 수 있음. 시점을 고려한 판정
- **환각 vs 오기**: LLM이 생성한 텍스트의 환각과, 인간이 작성한 문서의 오기는 다름. 이 스킬은 후자(인간 작성 문서의 오기)를 탐지
